I simplified the code removing parentheses and the times ballooned.
The code below writes text to the screen and to nul and appends to a file, as you would do in the real world.

Is anyone else willing to compare their machines and OS with this version?

Code: Select all

output to screen: 1110 msec
output to nul:    1300 msec
output to file:   3500 msec



Code: Select all

@echo off
setlocal

set a=abc

del junk.txt 2>nul

set t1=%time%
  for /l %%N in (1 1 10000) do echo %a%

set t2=%time%
  for /l %%N in (1 1 10000) do echo %a%>nul

set t3=%time%
  for /l %%N in (1 1 10000) do echo %a%>>junk.txt

set t4=%time%

call :showTime "output to screen:" "%t1%" "%t2%"
call :showTime "output to nul:   " "%t2%" "%t3%"
call :showTime "output to file:  " "%t3%" "%t4%"

goto EOF

:showTime
setlocal
for /f "tokens=1-4 delims=:." %%d in (%2) do set /a "t1=%%d*360000+1%%e*6000+1%%f*100+1%%g-610100"
for /f "tokens=1-4 delims=:." %%d in (%3) do set /a "t2=%%d*360000+1%%e*6000+1%%f*100+1%%g-610100, t=t2-t1"
>>results.txt echo %~1 %t%0 msec
exit /b

The results you got when removing parentheses from my test case are totally expected. Outer parentheses redirection is executed once. Redirection within the loop is executed with each iteration, and it takes time to set up the redirection.

That is why in other threads we have recommended to use the outer parentheses with > redirection instead of using >> redirection within a loop.

--------------------------------------------------

Your results with my original test prove that writing to file / nul is much faster then writing to screen, even on your machine. So I haven't a clue as to why your machine acts the way it does when executing alan's original code. It certainly is evidence against my theory that writing an empty string to the screen takes time when using SET /P "VAR=".

Aacini wrote:Yes, this is the expected behavior. Yes, Dave Benham said something similar about what was expected.

BUT foxidrive has results that disprove this and show that it makes zero difference on his machine whether he uses > or >>

SUGGESTION :-

A file with 771 lines is read and written one line at a time.
771 lines are written :-
via ">" in 190 mSec by test X4
via ">>" in 990 mSec by test X8
If CMD.EXE writes a single line to a disc rotating at 7200 r.p.m., and has to wait for the data to be written to media,
it will have to wait 8.3 mSec for the next rotation before the next line gets written,
and 771 lines would take 6400 mSec, not the 990 mSec I observed.

However, CMD.EXE does not have to wait for each piece of data to be written after the disc has got into position,
instead Windows has RAM caches and HDD have RAM caches,
consequently as each of 771 lines is appended at approx 1.5 mSec intervals, Windows is NOT seeking the end of the file on the non-volatile disc,
because it will be somewhere in one of the caches, so Windows has a lot complex task.

An SSD does not require Windows caches.
What is sent to the SSD is immediately captured, and I assume is immediately available for reading back.
This MAY result in Windows facing a far less complex task when trying to append the next line to the previous.

I think I am one of many that still use Windows Caches even for the SSD, so the expected performance hit applies when I use >>

Perhaps there is no such performance hit when there is no such Windows Cache.

Could your secret ingredient be that you have no Windows cache applied to your SSD ?
